# ADR-001: Knuckles — Slug Coding Agent (V1 Local Agent, V2 GitHub SDLC Agent)

## Status

Proposed

## Context

Slug is being designed with strong introspection, explicit module structure, immutable state, and AI-friendly
manifests (`SLUG.ai`, `MANIFEST.ai`). These characteristics make it particularly suitable for building **AI-assisted
development tools** directly within the ecosystem.

The goal of **Knuckles** is to create a Slug-native coding agent that can:

* inspect repositories
* understand Slug modules and metadata
* propose and apply code changes
* validate modifications through tests and commands
* produce auditable change summaries

Early inspiration comes from minimal coding agents such as **Pi**, which emphasizes a small core toolset (read, write,
edit, run) and iterative reasoning over codebases.

Rather than building a general AI assistant, Knuckles focuses specifically on **software development workflows**.

The project is divided into two major phases:

### Version 1 — Local Coding Agent

A CLI-driven coding agent that operates on a local repository.

Primary capabilities:

* inspect source code
* propose patches
* apply edits
* run tests
* summarize results

This version focuses on **developer productivity** and experimentation.

### Version 2 — GitHub-Native SDLC Agent

A GitHub-integrated agent that operates within isolated environments and participates directly in a repository's
software development lifecycle.

Capabilities include:

* issue-driven development
* branch creation
* pull request generation
* test execution
* automated summaries
* guarded self-modification via PRs

This version transforms Knuckles into a **software worker operating under SDLC discipline**.

A key design goal is that **self-modification is not special**. Knuckles should improve its own codebase through the
same process it uses for any other repository.

---

# Decision

Knuckles will be implemented as a **structured coding agent framework in Slug**, with a modular architecture that
supports both local development workflows (V1) and GitHub-driven SDLC workflows (V2).

The architecture emphasizes:

* explicit state
* immutable transformations
* deterministic tool execution
* isolated execution environments
* auditable outputs

## 1. Core Design Principles

Knuckles follows several foundational design principles:

### Small Tool Surface

The agent interacts with repositories through a small set of explicit tools:

* read files
* list directories
* search source
* write files
* apply edits
* run commands
* inspect repository state

Reducing tool complexity improves determinism, safety, and debuggability.

### Immutable Session State

Agent execution is modeled as transformations over immutable session objects.

Each step produces a new session state containing:

* conversation history
* discovered facts
* pending tasks
* tool outputs
* proposed patches

This enables replay, debugging, and potential session branching.

### Explicit Planning

The agent does not directly execute free-form instructions.

Instead it:

1. interprets the task
2. produces structured actions
3. executes tools
4. evaluates results
5. repeats until completion

### Auditable Outputs

All changes must produce structured artifacts:

* patches
* execution logs
* summaries
* validation results

These artifacts allow developers to audit agent decisions.

### Repository-Aware Reasoning

Knuckles is designed to leverage Slug-specific metadata:

* `SLUG.ai`
* `MANIFEST.ai`
* `slug.meta.describe`
* module metadata
* parameter tags and documentation

This provides the agent with **structured language knowledge** unavailable to generic coding agents.

# Version 1 — Local Coding Agent

V1 implements a CLI-based coding assistant operating directly on a local repository.

## V1 Goals

* assist developers during coding
* analyze repositories
* propose targeted edits
* validate changes locally
* summarize results

The agent runs in the developer environment and does not integrate with external services.

## V1 Execution Loop

The agent operates through an iterative planning loop.

```
session
  -> prompt.build(session)
  -> model.complete(...)
  -> plan.parse(response)
  -> match action {
       toolCall => tool.execute(session, toolCall) |> run
       done     => finalize(session)
       error    => recover(session) |> run
     }
```

Each iteration produces a new immutable session.

## V1 Tool Interface

The minimal tool set includes:

```
read(path)
list(directory)
find(pattern)
write(path, content)
edit(path, search, replace)
run(command)
diff()
```

These tools are sufficient for implementing most code modifications.

## V1 Session Model

A session represents a single coding task.

Example structure:

```
@struct(Session)
{
  goal: @str,
  repoRoot: @str,
  history: @list,
  facts: @list,
  todos: @list,
  openFiles: @list,
  patches: @list,
  results: @list,
}
```

This state evolves during execution.

## V1 Patch Model

Changes are applied through patch proposals.

Workflow:

1. agent proposes patch
2. runtime validates file targets
3. patch is applied
4. validation commands run
5. results recorded

Direct uncontrolled file modification is avoided.

## V1 CLI Interface

Example usage:

```
slug run knuckles.agent \
  --repo . \
  --goal "Add caching to evaluator and update tests"
```

The agent analyzes the repository and proposes modifications.

# Version 2 — GitHub SDLC Agent

V2 extends Knuckles to operate directly within repository development workflows.

The agent becomes a **GitHub-native SDLC participant**.

## V2 Goals

* automate routine development tasks
* operate through issues and pull requests
* run in isolated environments
* produce reviewable changes
* support controlled self-improvement

## V2 Work Items

The unit of work becomes a structured task derived from GitHub activity.

Example sources:

* issue assignment
* PR review
* failing CI pipeline
* scheduled maintenance
* repository housekeeping

Work items are represented explicitly.

```
@struct(WorkItem)
{
  id: @str,
  kind: @symbol,
  repo: @str,
  baseBranch: @str,
  title: @str,
  body: @str,
  acceptance: @list,
  constraints: @list,
}
```

## V2 Architecture

V2 introduces several new system layers.

### Control Plane

Responsible for:

* receiving work items
* scheduling runs
* creating isolated workspaces
* tracking run results
* publishing status

### GitHub Adapter

Encapsulates GitHub API operations.

Capabilities include:

* reading issues
* cloning repositories
* creating branches
* committing files
* opening pull requests
* posting comments
* inspecting CI checks

### Workspace Runtime

Each run executes in an isolated environment.

Properties:

* ephemeral filesystem
* fresh repository checkout
* restricted tool access
* configurable CPU/time limits
* controlled network access

Isolation improves safety and reproducibility.

### Agent Core

The V1 reasoning loop is reused, but enhanced with SDLC awareness:

* issue interpretation
* repository conventions
* acceptance criteria
* branch strategies
* PR quality standards

### Policy Layer

Defines operational constraints:

* repositories allowed
* commands permitted
* branch protections
* diff limits
* secret access
* self-modification permissions

Policies ensure predictable and safe operation.

# V2 SDLC Workflow

The primary workflow is issue-driven development.

1. issue assigned to Knuckles
2. agent analyzes repository and issue context
3. agent creates feature branch
4. agent implements change
5. agent runs validation commands
6. agent opens pull request

PR descriptions include:

* goal
* implementation summary
* files changed
* validation performed
* risks
* follow-up work

This ensures high-quality artifacts.

---

# Self-Modification Model

Knuckles may eventually modify its own repository.

Self-modification follows the same SDLC process as any other project.

Levels of autonomy:

### Level 0

Knuckles cannot modify its own repository.

### Level 1

Knuckles may propose changes to itself via pull requests.

Human approval required.

### Level 2

Knuckles may modify non-critical components:

* prompts
* manifests
* documentation
* tests

Still requires review.

### Level 3

Knuckles may merge low-risk changes automatically under policy constraints.

Initial implementation targets **Level 1** only.

# Slug-Specific Advantages

Knuckles leverages several Slug features that improve agent reasoning.

### SLUG.ai

Defines the language grammar and core capabilities in an AI-optimized format.

### MANIFEST.ai

Provides machine-readable module descriptions for libraries.

### slug.meta.describe

Enables runtime introspection of:

* modules
* functions
* parameters
* tags
* documentation

These capabilities allow Knuckles to analyze Slug code with significantly more structure than generic code agents.

# Consequences

## Positive

* Enables Slug-native AI development workflows
* Creates structured coding automation
* Improves developer productivity
* Leverages Slug’s strong metadata system
* Provides a safe path toward self-improving development agents
* Produces auditable and reproducible change artifacts

## Negative

* Requires careful sandbox design
* GitHub integration introduces operational complexity
* agent reliability depends on strong prompt discipline
* execution costs increase with isolated environments

## Neutral

* agent capabilities depend on available language model providers
* future improvements may require additional tooling layers
* policy systems will evolve as capabilities expand
